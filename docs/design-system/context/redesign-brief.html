Redesign Brief

Standing context for this project — read first, every session. APEX today is a React Native strength-training app; this document is the brief for rebuilding it native in SwiftUI with a full visual and interaction redesign. Because this page documents the app as it exists today, the palette in this page's :root is today's production palette, not a candidate direction.

1What this is

APEX is being rebuilt native — SwiftUI, targeting iOS 26 — with a full visual and flow redesign, not a like-for-like port. This claude.ai/design project is the design hub for that effort.

The pipeline has three stages, each landing in its own part of this project: art-direction rounds are pushed as candidate pages into specimens/; once a direction is locked, it's codified into foundations/ (colors, typography, spacing, core components); every subsequent screen is then designed here against that locked foundation.

Mockups produced in Claude Design are exported, converted to static HTML, and committed to docs/mockups/ in the app repo. That folder — not this project, not memory — remains the implementation contract that engineering builds against.

2The bar

Production-grade, lovable, Ive-grade restraint. The product's soul is minimal touches to complete a goal — a set logged exactly as prescribed should take one tap, nothing more.

Binding rule: every redesigned flow must reduce or hold its interaction count versus today. It may never increase it. A redesign that looks cleaner but takes more taps has failed the bar.

3What stays

  • Dark-first, instrument-panel DNA — the app reads like a cockpit display, not a consumer social app.
  • The APEX name and the ΛPEX wordmark, set in Orbitron 800.
  • Uppercase eyebrow labels over large numerals — the app's core typographic signature for stats and metrics.
  • Hairline-bordered flat cards as the primary content container.
  • No vanity metrics — every number shown must be actionable or tell a story.
  • State is sacred — in-progress sessions survive app kills and restore to the exact position on relaunch.

4What changes

  • The indigo #6366f1 accent — today's --accent, shown throughout this page's own styling — is retired. A new accent is one of the first things a candidate direction proposes.
  • Cleaner and more minimal throughout — fewer competing visual elements per screen.
  • Typography refined — weights, scale, and rhythm revisited, not just recolored.
  • A specific list of interaction flows get fixed as part of the redesign, not left as-is — see the next section.

5Flow fixes to design in

Each of these is a known problem in the current app, carried in as a requirement for the corresponding redesigned flow — not an afterthought to patch later.

  • #66Workout logging — a mis-tap collapses the row; there's no way to add or remove sets; adding/reordering exercises is janky; tap targets are too small.
  • #70Session memory — an in-progress session must surface on Home, not just be recoverable from inside Workout.
  • #46Warmup protocols — warmup steps need to be expandable, not a flat undifferentiated list.
  • #68PRs & trends — personal records need to be viewable, and trend charts need to be readable, not decorative.
  • #67Keyboard handling — the keyboard must never cover note input fields.
  • #65Session timestamps — session timestamps must be editable after the fact.
  • #69Save prompt — kill the redundant post-workout save prompt.

6The Liquid Glass question

Open — answer per direction

iOS 26 introduces Liquid Glass, a system-wide material language. APEX's stance on it is an open question each candidate direction must answer explicitly, not by accident — and it should be answered separately for three surfaces:

  • Tab bar — embrace, resist, or blend?
  • Sheets (modals) — embrace, resist, or blend?
  • Cards — embrace, resist, or blend?

The three answers need not match — a direction might, for example, embrace native glass materials for the tab bar and sheets while keeping cards flat and hairline-bordered to preserve the instrument-panel read.

7Constraints

  • SwiftUI / iOS 26 native — Swift Charts for all charts, Dynamic Type support throughout, .sheet for modal presentation.
  • Dark-only for v1 — no light mode to design or support.
  • Block color-coding must survive in some form — the four training-block categories (hypertrophy, strength, deload, realization) are today color-coded
    and that distinction needs to carry forward, even if the specific hues change.
  • Running keeps a distinct accent — the supplementary-goal tab reads as visually separate from the core lifting accent.
  • The 4-step text ramp and semantic good / warn / bad colors continue as concepts — the specific hex values are open, but the structure (a four-level text-emphasis hierarchy; green/amber/red semantics for good/warn/bad states) is not.

8How rounds work

Each round, Ben reacts to every specimen individually — keep, kill, or blend — with notes, in claude.ai/design.

Those reactions produce round N+1: killed directions drop, kept ones continue, blended ones combine. There is no cap on the number of rounds — this runs until the direction is right, not until a round budget is spent.

Exit criterion: Ben declares a direction locked. That single act triggers foundations codification (colors, typography, spacing, core components written into foundations/) and generation of the canonical tokens.json, which every downstream consumer — the Swift app, the shared TS theme, this design-system project itself — then generates from.
